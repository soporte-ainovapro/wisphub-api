# Authentication

La WispHub API implementa **dos capas de autenticación** independientes:

1. **JWT Bearer Token** — protege todos los endpoints de esta API. Los clientes (bots, frontends, etc.) deben obtener un token antes de consumir cualquier recurso.
2. **WispHub Net API Key** — usada internamente por la API para autenticarse con el backend de WispHub Net. Los clientes de esta API **no necesitan** gestionar esta clave.

---

## Part 1: JWT Authentication (API Clients)

### Authentication Flow

```
┌──────────────┐
│ Client App   │
│ (Bot/Web)    │
└──────┬───────┘
       │
       │ POST /api/v1/auth/token
       │ {username, password}
       ▼
┌──────────────────┐
│  WispHub API     │  Valida credenciales → emite JWT
│  (This API)      │
└──────┬───────────┘
       │
       │ Authorization: Bearer <jwt_token>
       │ (todas las solicitudes posteriores)
       ▼
┌──────────────────┐
│  Endpoints       │  get_current_user dependency
│  Protegidos      │  valida y decodifica el JWT
└──────┬───────────┘
       │ (si el JWT es válido)
       │ HTTP Request
       │ Header: Authorization: Api-Key {KEY}
       ▼
┌──────────────────┐
│  WispHub Net     │  API Key validation
│  (External)      │
└──────────────────┘
```

### Obtaining a Token

Llama al endpoint de login con tus credenciales como **form-data**:

```bash
curl -X POST http://localhost:8000/api/v1/auth/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "username=admin&password=your_password"
```

Respuesta:

```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "bearer"
}
```

<Info>
  El endpoint usa el estándar OAuth2 `application/x-www-form-urlencoded`, no JSON. Los campos son `username` y `password`.
</Info>

### Using the Token

Incluye el token en el header `Authorization` de **todas** las solicitudes a endpoints protegidos:

```
Authorization: Bearer <access_token>
```

**Ejemplo:**

```bash
curl http://localhost:8000/api/v1/clients/ \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
```

### Token Properties

| Property | Value |
|---|---|
| Algorithm | HS256 |
| Default expiry | 60 minutes (configurable) |
| Claim | `sub` = username |
| Header scheme | `Bearer` |

### Token Expiry

Cuando el token expira, los endpoints protegidos devuelven:

```json
{
  "detail": "Could not validate credentials."
}
```

Con el header `WWW-Authenticate: Bearer`. Simplemente solicita un nuevo token.

### JWT Configuration

Las siguientes variables de entorno controlan el comportamiento del JWT:

```env .env
# Clave secreta para firmar los tokens — usa un valor aleatorio largo
JWT_SECRET_KEY=your_very_long_random_secret_key_here

# Algoritmo de firma (por defecto HS256)
JWT_ALGORITHM=HS256

# Minutos de validez del token (por defecto 60)
JWT_ACCESS_TOKEN_EXPIRE_MINUTES=60

# Credenciales de acceso a la API
API_USERNAME=admin
API_PASSWORD_HASH=$2b$12$...  # hash bcrypt de tu contraseña
```

<Warning>
  `JWT_SECRET_KEY` debe ser una cadena aleatoria larga y segura. Nunca uses valores predecibles como `"secret"` o `"1234"` en producción. Genera una con: `openssl rand -hex 32`
</Warning>

<Tip>
  Para generar el hash bcrypt de tu contraseña, usa Python:
  ```python
  from passlib.context import CryptContext
  pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")
  print(pwd_context.hash("tu_contraseña"))
  ```
</Tip>

### How JWT is Implemented

La verificación del token en los endpoints se hace mediante una FastAPI dependency:

```python app/api/dependencies.py
from fastapi import Depends
from fastapi.security import OAuth2PasswordBearer
from app.core.security import decode_access_token

oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/api/v1/auth/token")

async def get_current_user(token: str = Depends(oauth2_scheme)) -> str:
    return decode_access_token(token)
```

Todos los endpoints protegidos incluyen `_: str = Depends(get_current_user)` en su firma, lo que hace que FastAPI rechace automáticamente solicitudes sin token válido con HTTP 401.

## Part 2: WispHub Net API Key (Outbound)

### Authentication Flow

```
┌──────────────┐
│ Client App   │
│ (Bot/Web)    │
└──────┬───────┘
       │
       │ HTTP Request
       │ Header: Authorization: Bearer {JWT}
       ▼
┌──────────────────┐
│  WispHub API     │  JWT validation
│  (This API)      │  (get_current_user)
└──────┬───────────┘
       │
       │ HTTP Request
       │ Header: Authorization: Api-Key {KEY}
       ▼
┌──────────────────┐
│  WispHub Net     │  API Key validation
│  (External)      │
└──────────────────┘
```

<Info>
  Los clientes se autentican con esta API usando JWT (ver Part 1). La API actúa como proxy de confianza hacia WispHub Net usando su propia Api-Key, sin exponerla a los clientes.
</Info>

## Configuration

### Environment Variables

Authentication credentials are configured via environment variables:

```env .env
WISPHUB_NET_KEY=your_api_key_here
WISPHUB_NET_HOST=https://api.wisphub.net
```

<Warning>
Never commit the `.env` file to version control. Add it to `.gitignore` to prevent credential leakage.
</Warning>

### Settings Management

Credentials are loaded using Pydantic Settings:

```python app/core/config.py
from pydantic_settings import BaseSettings, SettingsConfigDict

class Settings(BaseSettings):
    model_config = SettingsConfigDict(env_file=".env", case_sensitive=True)

    # WispHub Net authentication
    WISPHUB_NET_KEY: str
    WISPHUB_NET_HOST: str
    
    # ... other settings

settings = Settings()
```

<Tip>
The `case_sensitive=True` setting ensures environment variable names must match exactly (e.g., `WISPHUB_NET_KEY`, not `wisphub_net_key`).
</Tip>

## Authentication Headers

All requests to WispHub Net include the API key in the `Authorization` header:

```python app/services/clients_service.py
HEADERS = {
    "Authorization": f"Api-Key {settings.WISPHUB_NET_KEY}"
}

async def fetch_client(params: Dict[str, str]) -> Optional[ClientResponse]:
    async with httpx.AsyncClient(timeout=10) as client:
        response = await client.get(
            settings.CLIENTS_URL,
            headers=HEADERS,
            params=params
        )
```

The format is:
```
Authorization: Api-Key <your_api_key>
```

## Security Best Practices

<Steps>
  <Step title="Use Environment Variables">
    Never hardcode API keys in source code. Always use environment variables:
    
    ```python
    # ❌ NEVER DO THIS
    API_KEY = "abc123def456"
    
    # ✅ CORRECT
    API_KEY = os.getenv("WISPHUB_NET_KEY")
    ```
  </Step>
  
  <Step title="Restrict File Permissions">
    Ensure the `.env` file has restricted permissions:
    
    ```bash
    chmod 600 .env
    ```
    
    This makes it readable only by the file owner.
  </Step>
  
  <Step title="Use Secrets Management in Production">
    For production deployments, use proper secrets management:
    
    - **Docker**: Use Docker secrets or environment variables
    - **Kubernetes**: Use Kubernetes secrets
    - **Cloud**: Use AWS Secrets Manager, Azure Key Vault, or Google Secret Manager
  </Step>
  
  <Step title="Rotate Keys Regularly">
    Establish a key rotation policy:
    
    - Generate new API keys quarterly
    - Invalidate old keys after migration
    - Maintain audit log of key usage
  </Step>
  
  <Step title="Monitor for Unauthorized Access">
    Watch WispHub Net logs for:
    
    - Unexpected API usage patterns
    - Failed authentication attempts
    - Requests from unknown IP addresses
  </Step>
</Steps>

## Docker Deployment

When deploying with Docker, pass environment variables securely:

### Using --env-file (Development)

```bash
docker run -d \
  --name wisphub_api_server \
  -p 8000:8000 \
  --env-file .env \
  wisphubapi:latest
```

### Using Docker Secrets (Production)

```yaml docker-compose.yml
version: '3.8'

services:
  api:
    image: wisphubapi:latest
    ports:
      - "8000:8000"
    secrets:
      - wisphub_api_key
    environment:
      WISPHUB_NET_KEY_FILE: /run/secrets/wisphub_api_key
      WISPHUB_NET_HOST: https://api.wisphub.net

secrets:
  wisphub_api_key:
    external: true
```

Then create the secret:

```bash
echo "your_api_key" | docker secret create wisphub_api_key -
```

## Error Handling

### Invalid API Key

If the API key is invalid, WispHub Net returns a 401 or 403 error:

```python
if response.status_code in [401, 403]:
    # Authentication failed
    return None
```

The middleware API handles this gracefully:

```json
{
  "ok": false,
  "type": "error",
  "action": "CLIENT_NOT_FOUND",
  "data": null,
  "message": "Unable to fetch client data"
}
```

<Warning>
Authentication failures are logged but not exposed to end users to prevent information disclosure.
</Warning>

### Missing API Key

If the `WISPHUB_NET_KEY` environment variable is not set, the application will fail to start:

```python
from pydantic import ValidationError

try:
    settings = Settings()
except ValidationError as e:
    print(f"Configuration error: {e}")
    sys.exit(1)
```

Error output:
```
Configuration error: 1 validation error for Settings
WISPHUB_NET_KEY
  field required (type=value_error.missing)
```

## API Key Permissions

The WispHub API key must have the following permissions in WispHub Net:

<Accordion title="Required Permissions">
- **Read Clients** (`api/clientes/`): View client information
- **Update Clients** (`api/clientes/{id}/perfil/`): Update client profiles
- **Read Plans** (`api/plan-internet/`): View internet plans
- **Read/Write Tickets** (`api/tickets/`): Create and view support tickets
- **Read Tasks** (`api/tasks/`): View task information
</Accordion>

<Warning>
Use the principle of least privilege. Only grant permissions that the API actually needs.
</Warning>

## Rate Limiting

WispHub Net may impose rate limits on API key usage:

- **Typical limit**: 1000 requests per hour
- **Burst limit**: 100 requests per minute

The caching system helps avoid rate limits:

- Without cache: ~6,000 requests/hour for 100 users
- With cache: ~12 requests/hour for 100 users

<Info>
The 5-minute cache TTL for clients and 15-minute TTL for plans effectively keeps the API well under typical rate limits.
</Info>

## Troubleshooting

### Issue: "401 Unauthorized" errors

**Cause**: Invalid or expired API key

**Solution**:
1. Verify the API key in your `.env` file
2. Check that the key hasn't been revoked in WispHub Net
3. Ensure no extra whitespace in the environment variable

```bash
# Check the loaded value
docker exec wisphub_api_server env | grep WISPHUB_NET_KEY
```

### Issue: "403 Forbidden" errors

**Cause**: API key lacks required permissions

**Solution**: Contact your WispHub administrator to grant the necessary permissions to your API key.

### Issue: Application won't start

**Cause**: Missing `WISPHUB_NET_KEY` environment variable

**Solution**: Ensure the `.env` file exists and contains the required variable:

```bash
cat .env | grep WISPHUB_NET_KEY
```
