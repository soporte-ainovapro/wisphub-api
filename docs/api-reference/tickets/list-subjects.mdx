# List Ticket Subjects

Returns all valid subjects for creating support tickets in WispHub, grouped by priority level. Ideal for bots or external interfaces that need to present the user with a list of options before opening a support request.

## Endpoint

```
GET /api/v1/tickets/subjects
```

<Info>
  This endpoint requires JWT authentication. Include the `Authorization: Bearer <token>` header.
</Info>

## Request

No parameters or request body required.

### Headers

| Header | Value | Required |
|---|---|---|
| `Authorization` | `Bearer <access_token>` | ✅ Yes |

### Response Fields

<ResponseField name="data.by_priority" type="object">
  Subjects grouped by priority level. Useful for building categorized dropdown menus.

  <Expandable title="priority levels">
    <ResponseField name="low" type="string[]">
      Priority level 1: administrative or non-urgent requests.
    </ResponseField>
    <ResponseField name="normal" type="string[]">
      Priority level 2: service degradation issues (slowness, intermittent connection).
    </ResponseField>
    <ResponseField name="high" type="string[]">
      Priority level 3: complete service loss or equipment damage.
    </ResponseField>
    <ResponseField name="very_high" type="string[]">
      Priority level 4: trunk infrastructure or fiber optic failures.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="data.all" type="string[]">
  Flat list of all valid subjects regardless of priority. Useful for validation or search.
</ResponseField>

## Priority Reference Table

| Priority Level | Value | Description |
|---|---|---|
| `low` | 1 | Administrative requests, no technical urgency |
| `normal` | 2 | Service degradation (slowness, intermittent connectivity) |
| `high` | 3 | Complete service loss or damaged equipment |
| `very_high` | 4 | Trunk infrastructure or fiber optic failure |

<Tip>
  The priority level determines how urgently WispHub dispatches a technician. Use `very_high` only for large-scale infrastructure failures.
</Tip>

## Code Examples

<CodeGroup>

```bash curl
curl -X GET http://localhost:8000/api/v1/tickets/subjects \
  -H "Authorization: Bearer <your_token>"
```

```python httpx
import httpx

async def get_ticket_subjects(token: str) -> dict:
    async with httpx.AsyncClient() as client:
        response = await client.get(
            "http://localhost:8000/api/v1/tickets/subjects",
            headers={"Authorization": f"Bearer {token}"},
        )
        response.raise_for_status()
        data = response.json()["data"]
        return data

# Get only high-priority subjects
subjects = await get_ticket_subjects(token)
high_priority = subjects["by_priority"]["high"]
```

```javascript fetch
async function getTicketSubjects(token) {
  const res = await fetch("http://localhost:8000/api/v1/tickets/subjects", {
    headers: { Authorization: `Bearer ${token}` },
  });
  const { data } = await res.json();
  return data;
}

// Usage
const subjects = await getTicketSubjects(token);
console.log(subjects.by_priority.high); // high-priority subjects array
console.log(subjects.all);              // all subjects flat list
```

</CodeGroup>

## Use Case: Bot Integration

This endpoint is especially useful for WhatsApp bots or self-service interfaces:

1. **When starting a support flow**, call `GET /api/v1/tickets/subjects` to fetch the up-to-date list.
2. **Present the options to the user** grouped by urgency or as a flat list.
3. **When creating the ticket**, pass the selected subject to `POST /api/v1/tickets`. WispHub will automatically assign the priority based on the subject.
